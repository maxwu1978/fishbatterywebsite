# Reel Mate 客服实施方案

更新日期：`2026-04-05`

## 目标

客服方案的目标不是做一个看起来很先进的聊天框，而是先确保：

1. 不漏掉任何一条询盘
2. 用户在付款前能快速获得真人确认
3. 页面上的客服能力和真实线上能力一致

## 当前建议路线

### 第一阶段：表单咨询 + Slack 通知

这是当前推荐的正式方案。

原因：

- 开发量最小
- 最符合高客单价、询盘型销售节奏
- 比半成品 AI 聊天更可靠
- 便于后续接入 Crisp 或 AI，而不需要推翻当前流程

当前要点：

- 使用 `api/inquiry.js` 作为统一入口
- 英文页、日文页、批发页都走同一个询盘接口
- 每条询盘推送到 Slack webhook
- 没有配置 webhook 时，询盘仍成功，但只写入 Vercel logs

### 第二阶段：Crisp 真人聊天

适合在满足以下条件后再启用：

- 询盘量开始上升
- 有人能在工作时段盯消息
- 常见问题已经足够清晰，不会把真人聊天变成重复解释基础信息

启用方式：

- 在 `chat-config.js` 中填写 `crispWebsiteId`
- 先只启用真人聊天，不启用自动 AI 首答
- 继续保留表单页，作为长问题和非即时问题入口

### 第三阶段：AI 首答 + 人工接管

这是最后阶段，不是当前主线。

前提：

- 知识边界足够明确
- 兼容性、配送、保证、付款口径都已稳定
- 已经有 Crisp 或人工流程作为接管出口

否则风险：

- AI 把兼容性说过头
- AI 把物流和售后承诺说死
- 用户误以为机器人回答等于正式承诺

## 页面入口策略

### 1. 首页

定位：

- 轻量预售前咨询入口

应该承接的问题：

- 是否适合我的电动轮
- 发到哪个国家
- 套装里包含什么
- 付款前能否先确认

### 2. `contact.html`

定位：

- 英文正式支持表单页

说明：

- 不再假装“即时 AI 聊天”
- 重点是结构化收集问题，方便客服快速回

### 3. `ja-contact.html`

定位：

- 日文正式支持表单页

说明：

- 适合承接日本用户的适配、配送、支付和充电器相关问题

### 4. `wholesale.html`

定位：

- 渠道、批发、OEM / ODM 入口

## Slack 通知建议

推荐每条通知至少包含：

- inquiry type
- name
- country
- source page
- topic
- reel details
- message

当前实现方向：

- 已在 `api/inquiry.js` 中增强为更适合 Slack 的消息结构
- 后续只需配置 `INQUIRY_WEBHOOK_URL`

## 明确的边界

以下结论必须保持一致，避免跑偏：

- 当前生产环境客服 = 表单咨询
- 当前生产环境不等于 live chat
- 当前生产环境不等于 AI chat
- `/api/chat` 当前不是 Vercel 线上能力
- Crisp 只有在填入 `crispWebsiteId` 后才算正式启用

## 当前推荐下一步

1. 配置 `INQUIRY_WEBHOOK_URL`
2. 在 Slack 验证英文页、日文页、批发页的询盘通知是否都能收到
3. 继续保持表单为主入口
4. 等询盘量上来后，再决定是否启用 Crisp 真人聊天

## 一句话结论

当前最好的客服方案不是“马上上 AI”，而是先把 `统一表单 + Slack 通知 + 清晰人工响应` 跑稳，然后再考虑 Crisp，最后才考虑 AI 首答。
